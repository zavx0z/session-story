# Session Story Pipeline

`session-story` — это конвейер превращения рабочей AI-сессии в человеческую историю.

Проект должен мыслиться не как утилита для кадров, а как связка этапов:

```text
recording
  -> video edit
  -> frame preparation
  -> visual analysis
  -> timeline merge
  -> keyframes
  -> narrative handoff
  -> voiceover / story / publication
```

## Зачем это нужно

Рабочая AI-сессия сама по себе непонятна обычному зрителю.

На экране могут быть:

- терминал;
- браузер;
- Codex;
- код;
- локальный сервер;
- отладчик;
- UI, который меняется по ходу работы;
- паузы, ожидание, тесты, ошибки и исправления.

Для разработчика это может быть понятно, но для обычного человека нужно собрать историю:

- что происходит;
- почему это важно;
- где человек задаёт смысл;
- где ИИ выполняет работу;
- где видно, что ИИ уже управляет рабочей средой;
- какой результат получается.

## Главная формула

```text
AI делает работу.
Видео фиксирует процесс.
Qwen видит, что было на экране.
GPT объясняет, почему это важно.
ai-macos автоматизирует рутину между ними.
```

## Роли компонентов

### Video processing

Первый слой работает с сырой записью экрана.

Цели:

- убрать простои;
- ускорить технические участки;
- вырезать лишние интервалы;
- получить короткое видео, которое можно смотреть;
- сохранить монтажные решения воспроизводимо.

### Frame preparation

Второй слой превращает финальное видео в кадры с таймкодами.

`prepare` уже делает:

- извлечение кадров;
- наложение таймкода;
- переименование кадров с временем;
- batch-и;
- contact sheets;
- prompt для Qwen.

### Visual analysis

Третий слой — Qwen Vision.

Qwen должен смотреть кадры визуально и описывать:

- что видно на экране;
- что делает пользователь;
- что делает ИИ;
- где запускаются команды;
- где идёт ожидание;
- где изменяется интерфейс;
- какие кадры важны для истории.

Это не художественный сценарий. Это фактический visual timeline.

### Browser automation

Четвёртый слой — `ai-macos`.

Он нужен, чтобы не отправлять batch-и в Qwen вручную.

Будущий сценарий:

```text
session-story подготовил batch
  -> ai-macos открыл Qwen Studio
  -> загрузил contact sheet или кадры
  -> вставил prompt
  -> дождался ответа
  -> сохранил JSON в qwen-results
  -> перешёл к следующему batch
```

### Timeline merge

`merge` проверяет, что все batch-и обработаны, и собирает полный таймлайн.

Это важная гарантия: нельзя случайно сделать дикторку по половине видео.

### Keyframes

`keyframes` выбирает кадры, которые нужны GPT и человеку:

- начало задачи;
- важные визуальные доказательства;
- момент написания кода;
- запуск тестов;
- изменение UI;
- финальный результат.

### GPT handoff

Финальный слой — GPT.

GPT получает:

- полный фактический таймлайн;
- ключевые кадры;
- контекст проекта;
- при необходимости Codex session jsonl;
- целевую длительность ролика.

GPT делает:

- дикторский текст;
- драматургию;
- монтажные подсказки;
- описание ролика;
- формулировку для обычного зрителя.

## Что важно не потерять

Этот pipeline вырос из реального процесса:

1. была длинная запись около 40 минут;
2. простое ускорение оказалось грубым;
3. перешли к сравнению кадров и удалению статичных участков;
4. сделали ручные cut/speed операции;
5. получили короткое видео около 2:56;
6. попробовали OCR;
7. поняли, что OCR по всему экрану даёт мусор;
8. перешли к визуальному анализу через Qwen;
9. пришли к идее автоматизировать Qwen Studio через `ai-macos` на втором MacBook.

Это и есть правильный вектор проекта.
